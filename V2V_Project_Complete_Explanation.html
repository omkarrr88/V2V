V2V Blind Spot Detection (BSD) System
Complete Project Breakdown & Explanation

Overview: This project is a hybrid "Physics-AI" system that stops vehicles from crashing into each other when changing lanes. Instead of relying on cameras or radar (which can be blocked by other cars), it allows vehicles to talk to each other directly using radio signals. By sharing basic data (where they are, how fast they are going), the system mathematically predicts if a collision is about to happen and warns the driver.

1. The Problem: The Limits of Modern Sensors

Modern cars have sensors (radars, cameras) to monitor the blind spot — the area beside and slightly behind the car that the driver cannot see in the mirrors. However, these sensors require a direct "line-of-sight." In dense traffic (like Indian urban roads), a large truck might block the camera's view of a fast-moving car sneaking up behind the truck. When the driver attempts to change lanes, a catastrophic multi-car collision can happen.

2. The Solution: Core Concepts Explained

V2V Communication (Vehicle-to-Vehicle)

This is the ability of cars to send data directly to other cars nearby using radio waves. The advantage? Radio waves pass through obstacles. Even if a truck blocks the view, its radio signal travels right through the truck so the ego car (the driver's car) knows where everyone is.

BSM (Basic Safety Message)

The BSM is a standardized data packet that cars broadcast 10 times per second. Older V2V research relied on massive BSM packets needing 13 pieces of data (including turn signals, steering wheel angle, etc.).
Project Innovation: This project forces the system to run on only 5 basic fields: GPS X position, GPS Y position, Speed, Acceleration, and Braking. This means even old or cheap cars patched with a basic transmitter can use this system.

DSRC and C-V2X

These are the two competing radio technologies that allow V2V to work. DSRC is like Wi-Fi adapted for fast-moving cars (IEEE 802.11p). C-V2X uses cellular towers like 5G or 4G LTE. The project is designed to work with either.

Dead Reckoning (CA-CYR Model)

What happens if a radio signal is temporarily blocked by a large building? The system uses Dead Reckoning. Specifically, it uses the Constant-Acceleration Constant-Yaw-Rate (CA-CYR) model. It basically guesses: "The target car was moving forward at 40km/h and gradually turning right. Since I lost connection, I'll assume it continued doing exactly that for the last half-second."

3. The "CRI" Mathematical Engine (How it detects danger)

At the core of the project is the Collision Risk Index (CRI) V3.0. It calculates a risk score from 0 (Safe) to 1.0 (Critical) using three main pillars, combined through a "Severity Gate".

Pillar A: Deceleration Risk (R_decel)

This formula checks: Is there enough physical space for the car to hit the brakes and stop in time? It calculates the maximum possible braking force (based on road friction and air resistance), adds a 1.2-second human reaction time, and maps it against the actual gap between the cars. If the gap is smaller than the required stopping distance, the risk hits 1.0.

Pillar B: Time-to-Collision Risk (R_ttc)

This formula calculates the exact countdown until the two vehicles overlap in physical space based on their current speeds and acceleration. If the countdown drops below 4 seconds, it triggers a maximum risk score.

The Big Innovation: The "Severity Gate"

In older systems, if a car was a little bit close, a little bit fast, and had a little bit of GPS noise, the system would add those tiny risks together and trigger a false alarm (beeping continuously and annoying the driver).
This project uses a max(R_decel, R_ttc) severity gate. Mathematically, it multiplies the risk rather than adding it. Meaning: Unless at least ONE of the real physical threats (braking distance OR time-to-collision) crosses a dangerously high threshold independently, the system will silence the alarm.

4. System Architecture: The Four Layers

The system was built as a 4-layer data pipeline to mimic a real car network.

  • Layer 1 (Simulation): Uses Eclipse SUMO, a professional traffic simulation software, to generate thousands of realistic cars driving in Navi Mumbai (Atal Bridge). It injects dangerous scenarios like cars violating red lights.
  • Layer 2 (Communication): Simulates the radio waves bouncing in the air. It introduces deliberate GPS noise and packet loss (using the Gilbert-Elliott Markov model) to prove the system works even when sensors glitch.
  • Layer 3 (Intelligence Kernel): This is the brain. It takes the messy data and runs it through the CRI Physics equations and an AI model simultaneously.
  • Layer 4 (Analytics): A real-time visual dashboard (built with Streamlit) that plots the cars and risk alerts for the engineers to analyze.

5. Technologies and Frameworks Used

This project relies on a highly specialized mix of modern software tools:

Python 3.10+ Eclipse SUMO TraCI API XGBoost SMOTE Streamlit ROS2 ONNX

Technology What does it do in this project?
Eclipse SUMO & TraCI SUMO (Simulation of Urban MObility) is a German microscopic traffic simulator. It generates the digital cars. TraCI acts as the bridge that allows Python to grab data from SUMO frame-by-frame.
XGBoost A powerful Machine Learning algorithm. In this project, a dual Physics-AI architecture is used. The Physics CRI engine is the primary boss. However, XGBoost watches the data as a backup to look for weird hidden patterns of danger that strict math formulas might miss.
SMOTE Since only 2.27% of the simulated driving time resulted in near-miss crashes, the AI didn't have enough dangerous data to learn from. SMOTE (Synthetic Minority Over-sampling Technique) mathematically generates fake hazardous events during training so the AI learns how to spot critical danger.
Streamlit A hot Python web framework used to turn the massive background calculations into an attractive, interactive local dashboard.
ROS2 (Robot OS) A software used in actual autonomous cars (like Tesla/Waymo). A script in this project wraps the math logic inside ROS2 so it is immediately ready to be plugged into physical car hardware.
ONNX (Open Neural Network Exchange). It converts the XGBoost AI model into an ultra-fast raw format that can run instantly on tiny microchips inside car dashboard hardware.

6. Special Danger Testing (Scenarios)

To prove the math works, the engineers aggressively injected two hostile scenarios into the peaceful traffic simulation:

  • Traffic Signal Violation (TSV): Forces a random car entering an intersection to ignore a red light at full speed, creating an immediate cross-traffic T-bone danger.
  • Hilly Narrow Road (HNR): Forces the simulation to drop the road friction to 0.55 (like wet asphalt) and shrinks the lanes. This tests if the deceleration risk math correctly adapts to the fact that cars now take much longer to hit the brakes safely.

7. Summary of Success

The project was a massive success. It was tested across 146,051 recorded vehicle interactions over the simulated Atal Bridge geometry. By deploying the mathematical "severity gate", the Physics model achieved an incredibly high Area Under Curve (AUC) accuracy score of 0.9874. The system flawlessly rejected false alarms caused by temporary swerving, while the backup AI (XGBoost) confirmed an 81.8% detection rate for truly critical scenarios.
